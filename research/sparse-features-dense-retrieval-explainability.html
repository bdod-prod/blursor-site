Opening the Embedding Space to Inspection and Control

Dense retrievers are remarkably effective and almost completely opaque. A query goes in, a ranked list comes out, and the high-dimensional vectors that produced that ranking carry no labels, no explanations, no way to ask why document A beat document B. That opacity has been tolerable when retrieval just works — but it becomes a real problem when you need to debug a failure, audit a system for bias, or deliberately shift what gets surfaced.

Xetrieval attacks this from two directions at once. It first internalizes the kind of chain-of-thought reasoning that LLMs can generate about documents — compressing that signal into the embedding itself via a lightweight MLP that trains in one to two minutes. It then decomposes those enriched embeddings with a sparse autoencoder, producing a small set of human-readable features that explain why a document ranked where it did. The features aren't just descriptive: erasing the ones Xetrieval identifies as key causes the largest drops in query-document similarity, and amplifying them improves retrieval performance across multiple benchmarks.

The anchoring number: for the Snowflake retriever, the reasoning internalizer alone lifts NDCG@10 from 38.3 to 51.2 — recovering most of the 16.4-point gap that full chain-of-thought reasoning achieves at 54.7, without any inference-time LLM calls at scale.

The Problem: Dense Retrievers Are Opaque by Design

Dense retrieval works by mapping queries and documents into the same vector space, then ranking by cosine similarity. The geometry of that space encodes relevance, but it doesn't decompose into anything a human can inspect. There's no axis labeled "answers the question" or "matches the domain" — just 768 or 4096 floating-point numbers per document.

One approach to improving retrieval quality is to have an LLM reason about each document before embedding it — generating chain-of-thought explanations that surface latent relevance signals. This works well, but it scales linearly with corpus size. For a corpus of millions of documents, running an LLM over every one at query time is impractical. The cost doesn't go away just because the reasoning is useful.

Xetrieval's insight is that you don't need the LLM at inference time if you can teach the embedding model to approximate what the LLM would have said. A small MLP trained on LLM-generated reasoning embeddings can internalize that signal cheaply. Then a sparse autoencoder can pull apart the resulting embedding into interpretable components — features that activate on specific semantic patterns and can be named in plain language.

Architecture: Reasoning Internalizer + Sparse Autoencoder

The system has two stages. The first is a reasoning internalizer: a one-hidden-layer MLP with SiLU activation, trained via mean squared error to predict what a sentence embedding would look like if the document had been processed through LLM chain-of-thought reasoning. Three separate internalizers are trained — one each for Summary, Purpose, and QA aspects — on 11,796 StackExchange documents. Each takes about one to two minutes to train.

Training time per reasoning internalizer (vs. LLM CoT that scales linearly with corpus)
k=64
TopK sparsity level chosen for best mono-semanticity / retrieval-retention trade-off

The second stage is a TopK sparse autoencoder (k=64) trained on 84,860 documents. The SAE decomposes the enriched embeddings into sparse feature activations — at k=64, most features are zero for any given document, and the ones that fire are the ones doing the semantic work. That sparsity level was chosen because it best balances reconstruction error, mono-semanticity, and retrieval retention across the range of sparsity settings tested.

Once the SAE is trained, an automated pipeline uses an LLM to generate natural-language descriptions for each feature by examining the documents that activate it most strongly. The result is a feature dictionary: each entry has a name, a description, and a set of documents where it fires. When a query comes in, Xetrieval can identify which features overlap between the query and a retrieved document — and those overlapping features are the explanation for why the document ranked where it did.

The detection score evaluation — where an LLM classifies activating versus non-activating sentences given a feature hypothesis — shows that features from the reasoning-internalizer-augmented SAE are substantially more distinguishable than those from a raw SAE or a random SAE baseline. The features are coherent, not artifacts.

Retrieval Gains: Consistent but Bounded

Across 8 dense retrievers and 7 benchmarks, the reasoning internalizer consistently outperforms the base retriever and partially closes the gap to full chain-of-thought reasoning. The gains are real but not uniform — they depend heavily on how much room the base retriever has to improve.

+12.9 pts
NDCG@10 gain for Snowflake retriever with reasoning internalizer (38.3 → 51.2), recovering the majority of the CoT Reasoner's 16.4-point improvement

For weaker retrievers, the lift is substantial. Snowflake goes from 38.3 to 51.2 NDCG@10 with the internalizer, recovering most of the CoT gain. e5-large improves from 61.5 to 64.2, versus 66.5 for the full CoT reasoner. gte-base moves from 61.7 to 62.4, against a CoT ceiling of 64.4.

For already-strong retrievers, the picture is different. Qwen3-Embedding-4B starts at 69.2 NDCG@10 and actually drops slightly to 68.7 with the internalizer — the base model already encodes rich enough semantics that the additional reasoning signal adds noise rather than signal. This isn't a failure of the method so much as a ceiling effect: when a retriever already captures most of what the LLM would reason about, there's little left to internalize.

These results hold across six different LLM teachers — DeepSeek-R1, DeepSeek-V3, GPT-OSS-120B, Qwen3-32B, and others — confirming the approach isn't sensitive to which model generates the reasoning signal.

Causal Verification: Features That Actually Control Retrieval

The more important result isn't the retrieval improvement — it's the causal verification. A lot of interpretability work produces features that correlate with outcomes without actually driving them. Xetrieval tests this directly.

Largest drop Erasing Xetrieval key features → cosine similarity decreases most
Score increase Erasing non-overlap active features → similarity often rises (distracting content removed)
Smaller effect Steering non-key features → inconsistent, minor changes in retrieval rank

At the document level, erasing the features Xetrieval identifies as key produces the largest drops in query-document cosine similarity. Erasing non-overlapping active features — the ones that fire on the document but not the query — often increases similarity, suggesting those features encode content that's actively distracting from relevance. That's a meaningful finding on its own: some of what a document is "about" works against its retrieval score.

At the task level, amplifying key features improves retrieval performance on BRIGHT, ArguAna, and NQ. Suppressing them causes clear degradation. Steering non-key features produces smaller, less consistent changes. And across both levels, Xetrieval's features show stronger steering effects than features identified by a raw SAE — the reasoning internalizer isn't just improving retrieval quality, it's producing features that more precisely capture what drives retrieval decisions.

This matters because it closes the loop between explanation and control. The features aren't post-hoc rationalizations; they're causally upstream of the ranking. If you can identify them, you can intervene on them.

What to Do With This

The practical value here splits into two use cases. The first is debugging. If a retriever is surfacing the wrong documents for a query, Xetrieval gives you a way to ask why — which features fired, which ones overlapped with the query, which ones didn't. That's diagnostic information that dense retrieval has never offered before.

The second is steering. If you know which features drive retrieval for a particular task or domain, you can amplify them — either by adjusting how documents are embedded or by directly intervening in the feature space. The intervention experiments show this works, and the effects are larger when using Xetrieval's features than when using a raw SAE.

The main constraint to keep in mind: the gains from the reasoning internalizer are front-loaded toward weaker retrievers. If you're already running a strong embedding model like Qwen3-Embedding-4B, the retrieval improvement may be marginal or negative. The interpretability benefits still apply — the sparse features are still coherent and causally meaningful — but the case for adopting Xetrieval looks different depending on where your baseline sits.

For teams running mid-tier retrievers on large corpora where inference-time LLM reasoning is too expensive, Xetrieval offers a practical path: one to two minutes of training per aspect, no LLM calls at scale, and a feature dictionary that makes retrieval decisions auditable for the first time.

Key Takeaway

Xetrieval trains in minutes, requires no inference-time LLM calls at scale, and its sparse features pass causal intervention tests across 8 retrievers and 7 benchmarks.

Practitioners can now audit why a dense retriever ranked a document — and surgically steer retrieval outcomes — by identifying and manipulating a small set of interpretable sparse features in the embedding space.
